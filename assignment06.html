Web-Accessibility Project: California Science Center Website

DH150 2021 Summer - Project Documentation by Leah Kim

About the Project

This project is to test and improve the web-accessibility of museum websites. Specifically, I focused on improving the accessibility and SEO scores of the original California Science Center website. This was achieved through a multitude of steps, including semantic structure, color contrast, keyboard accessibility, screen reader, and ARIA labeling.

Research #

To start this project, I chose five different museum websites. I selected the websites of the American Museum of Natural History, The MET, California Science Center, MoMA, and the Louvre. Then, I utilized Google Chrome's development tool Lighthouse to run an audit on the accessibility and SEO scores of each website.

1) American Museum of Natural History https://www.amnh.org/, Accessibility (93), SEO (100)

2) The MET https://www.metmuseum.org/, Accessibility (91), SEO (83)

3) California Science Center https://californiasciencecenter.org/, Accessibility (93), SEO (75)

4) MoMA https://www.moma.org/, Accessibility (94), SEO (100)

5) The Louvre https://www.louvre.fr/en, Accessibility (100), SEO (100)

Examining the scores across these museum websites, I noticed some common accessibility and SEO issues that affected their respective scores. These included insufficient color contrast, links without descriptions, images without descriptions, lack of metadata, ARIA issues, and more. Through this investigation, I learned that accessibility and SEO are related, and having a higher SEO score typically improves accessibility. Case in point, many of the websites that had a high SEO score also generally had a high accessibility score. Some of the common issues that lowered accessibility scores were contrast issues, no discernible names for links, and navigation issues. One of the websites, the Louvre, had perfect accessibility and SEO scores, and while navigating the website, I found it very easy and clear to follow. The lowest SEO score was 75, which was for the California Science Center, and the lowest accessibility score was 91 for the MET. The California Science Center had a low SEO score because document does not have meta description, links do not have descriptive test, and links are not crawlable. This is the website I chose to improve. Overall, many of these websites were mostly accessible, but there is room for improvement to make it even more accessible. The Louvre's website is the model website.

Design Process #

Regarding the design of the website for both visual and accessibility purposes, I chose to improve the California Science Center website. This was because the accessibility score, although relatively high but lower compared to the others, had some room for improvement. As for the SEO score, it scored the lowest of the five museums, which was another reason for choosing this website. I also thought I could redesign the website to make it look cleaner and to make tabbing easier. The following highlights the steps I took to improve both the accessibility and SEO scores.

Semantic Structure

To ensure the semantic structure of HTML, I made sure to include all the components and double check that each one is placed in the correct order and that it is closed. This was fairly simple, but it was important as the first step to improve accessibility. The first parts of styling were also introduced in the forms of headings, line breaks, horizontal lines, and more.

Color Contrast

The following step was to ensure sufficient color contrast, at a ratio of 4.5:1. This required much more experimentation because it was not easy to utilize colors that fit the color scheme and had good contrast with each other. I used a color picker, a color contrast checker, and a color scheme design to incorporate some styling to my website. CSS was introduced, and I used this to add a font, change layouts, and more to improve the overall design.

Keyboard Accessibility

Afterwards, it was time to tackle keyboard accessibility. This was achieved through manual testing, and this was where the CSS features of hover and focus, as well as bypass blocks were important. The original website had slight issues with tabbing order, so I made sure to improve that on my own version. The manual testing of both websites helped to correct minor mistakes in keyboard accessibility.

Screen-reader and ARIA

In this final step, I added aria-labels to make sure the screen-reader can successfully identify all of the website components. This was also polished through testing with the screen-reader myself, and it seemed to be working well. Interactive icons for accessibility purposes, such as font-size increasing and contrast modes, were also included.

Through these accessibility-improving features, I learned that even simple steps can be taken to improve accessibility. The extra CSS design step can make a significant difference in improving website navigation, like the bypass blocks and hover options. Moreover, I learned how aria-labels and adding descriptive text is very important to accessibility. Many of these things are overlooked by those without disabilities, but after going through the process of designing a website with as much accessibility features as possible, I will always be looking out for these features on all websites. The original website had some insufficient color contrast according to the audit, and it also had a lack of metadata, links without discernible links, and links that were not crawlable. The scores were 93 and 75 respectively. My mockup website now has full accessiblility and SEO from all the improvements taken with each assignment. Particularly, the inclusion of metadata in the head of html and modifying link names made the biggest difference. In the future, I would like to work on adding interactive elements like a photo slideshow and more buttons for users. Overall, it was a great learning experience while building my accessible website.
